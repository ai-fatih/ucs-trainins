# Требования к модулю записи на консультации и обучение (rkeeper)

## 1. Цель проекта
Веб-приложение для оптимизации процессов отдела консультации и обучения по программе rkeeper:
автоматизация записи, коммуникации, оценки и аналитики.

---

## 2. Заинтересованные стороны (Stakeholders)

| Сторона | Роль | Потребности |
|---------|------|-------------|
| Клиент — юридическое лицо | Заказчик услуг | Запись сотрудников, видеть договоры, бесплатная поддержка |
| Клиент — физическое лицо | Пользователь | Самостоятельная запись, регистрация, далее тарифы |
| Специалисты отдела | Исполнители | Видеть расписание, вести консультации/тренинги |
| Администратор | Управление | Настройка услуг, слотов, прав, аналитика |
| ITSM-система | Внешний источник | Предоставление данных о договорах и компаниях |

## 5. Архитектурные решения (предварительно)

### 5.1. Две ветки пользователей

```
User (base)
├── CompanyUser (юрлицо)
│   ├── профиль из ITSM (sync)
│   ├── привязан к договору
│   └── сотрудники компании
└── IndividualUser (физлицо)
    ├── профиль в приложении
    ├── самостоятельная регистрация
    └── биллинг (post-MVP)
```

### 5.2. Event-Driven для уведомлений

```
Событие (запись создана, отменена, оценена)
  → Message Broker
    → Подписчики: Email, Telegram, SMS, WebSocket
```

### 5.3. Аналитическая схема

Отдельная read-only БД/схема для аналитики (CQRS-подход), чтобы отчёты не
нагружали основное приложение.

### 5.4. Безопасность (Security Architecture)

```
┌─────────────────────────────────────────────────────┐
│                   Internet                          │
├─────────────────────────────────────────────────────┤
│           WAF (Web Application Firewall)             │
├─────────────────────────────────────────────────────┤
│              API Gateway (NestJS)                    │
│  ┌───────────────────────────────────────────────┐  │
│  │  Middleware chain:                             │  │
│  │  Rate Limiter → Auth (JWT) → RBAC Guard →     │  │
│  │  Validation Pipe → Audit Logger → Controller  │  │
│  └───────────────────────────────────────────────┘  │
├─────────────────────────────────────────────────────┤
│              Service Layer                           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │ Auth     │ │ Booking  │ │ Chat     │  ...        │
│  │ Module   │ │ Module   │ │ Module   │            │
│  └──────────┘ └──────────┘ └──────────┘            │
├─────────────────────────────────────────────────────┤
│               Data Layer                             │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │PostgreSQL│ │  Redis   │ │ITSM Cache│            │
│  │ (РФ)     │ │ (РФ)     │ │(РФ)      │            │
│  └──────────┘ └──────────┘ └──────────┘            │
└─────────────────────────────────────────────────────┘
```

### 5.5. Интеграция с ITSM (Architecture)

```
Client Request → Cache (Redis, TTL 5 min)
  ├── Hit → Return cached data
  └── Miss → Call ITSM API → Update Cache → Return
       └── ITSM down → Return stale cache + warning header
                      → Log error → Alert admin
```

### 5.6. Data Flow (граждан РФ)

```
Все ПДн граждан РФ:
  Collection → Validation → Storage (PostgreSQL, Yandex Cloud Мск/Влад)
  Backup → Yandex Object Storage (РФ)
  No cross-border transfer without explicit consent + РКН notification
```

## 7. Вопросы к заказчику / риски

1. **ITSM**: какая система, какой API, есть ли документация? Есть ли тестовый контур?
2. **Услуги**: фиксированный список или администрируемый? Разная длительность? Групповые тренинги?
3. **Слоты**: специалист сам назначает или система рассчитывает из рабочих часов?
4. **Чат**: нужно ли соблюдать 149-ФЗ (ОРИ)? Есть ли юрист для консультации?
5. **Регистрация**: только email или ещё телефон? SMS-подтверждение обязательно?
6. **Инфраструктура**: какой облачный провайдер? Yandex Cloud, VK Cloud, Selectel?
7. **Брендинг**: есть ли гайдлайны rkeeper (цвета, шрифты, логотип)?
8. **Биллинг**: какая эквайринг-система? (post-MVP — ЮKassa, Tinkoff, Сбербанк)
9. **Язык**: только русский или мультиязычность?
10. **Сроки**: есть ли дедлайн по MVP?
11. **Команда**: кто реализует (фронт, бэк, дизайн)? Есть ли devops?
12. **Юрист**: есть ли доступ к юристу по 152-ФЗ для подготовки документов?
13. **ОРИ**: нужно ли регистрировать ОРИ (из-за чата) или чат считается внутренним?
14. **СОРМ**: если ОРИ — кто будет оплачивать и внедрять СОРМ?

## 8. Требования соответствия законодательству РФ (Compliance)

| Требование | Реализация |
