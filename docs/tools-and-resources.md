# Анализ инструментов и ресурсов

## 1. Технологический стек (с обоснованием)

### 1.1. Frontend — Next.js 14+ (App Router)

| Компонент | Инструмент | Зачем |
|-----------|-----------|-------|
| Framework | Next.js 14+ | SPA + SSR, файловая маршрутизация, RSC, API routes |
| Язык | TypeScript | Типобезопасность на всём стеке |
| UI | Tailwind CSS + shadcn/ui | Быстрая разработка, кастомизация, доступность |
| Client state | Zustand | Лёгкий, без бойлерплейта |
| Server state | TanStack Query (React Query) | Кэширование, инвалидация, оптимистичные обновления |
| Формы | React Hook Form + Zod | Валидация на клиенте и сервере |
| WebSocket | socket.io-client | Чат в реальном времени |
| Auth на клиенте | next-auth / Next.js middleware | JWT-токены, защита маршрутов |

**Почему Next.js:**
- Единый фреймворк для фронта и API (можно часть бэка вынести в API routes)
- App Router упрощает роутинг и layout
- Server Components снижают bundle-size
- Отличная DX и сообщество

### 1.2. Backend — NestJS

| Компонент | Инструмент | Зачем |
|-----------|-----------|-------|
| Framework | NestJS | Модульная архитектура, DI, декораторы |
| Язык | TypeScript | Единый тип с фронтом |
| ORM | Prisma | Type-safe доступ к БД, миграции, схемы |
| БД | PostgreSQL | Надёжность, JSONB для гибких полей |
| Кэш / брокер | Redis | Сессии, кэш ITSM-данных, pub/sub для уведомлений |
| WebSocket | @nestjs/platform-socket.io | Чат |
| Очереди | Bull + Redis | Фоновые задачи (уведомления, синхронизация ITSM) |
| Валидация | class-validator + class-transformer | DTO на сервере |
| Документация API | Swagger (OpenAPI) | Автодокументация |

**Почему NestJS:**
- Архитектура близка к enterprise (модули, сервисы, контроллеры)
- Встроенная поддержка WebSocket, очередей, авторизации
- Prisma + NestJS = лучшая связка TypeScript ORM
- Легко тестировать (DI, моки)

## 2. Ресурсы (что нужно подготовить)

### 2.1. Интеграция с ITSM

**Критический внешний ресурс.** Нужно от ITSM-команды получить:

1. **Документация REST API** (Swagger / Postman collection)
2. **Методы API:**
   - `POST /auth` — авторизация компании по логину/паролю или токену
   - `GET /contract/{id}` — проверка статуса договора
   - `GET /company/{id}` — данные компании (название, реквизиты)
   - `GET /company/{id}/employees` — список сотрудников
   - `POST /webhook` — уведомления об изменениях (опционально)
3. **Режим доступа**: service-account токен или OAuth2 Client Credentials
4. **Rate limits**: сколько запросов в минуту/час
5. **SLA**: время ответа, гарантии доступности
6. **Тестовый контур**: URL, тестовые договоры/компании

**Если ITSM-API нет или он не готов:** потребуется MVP-заглушка (mock-сервер)
с той же спецификацией, чтобы разработка не блокировалась.

## 3. Риски и их митигация

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| ITSM API не готов или отсутствует | Средняя | Высокое | Mock-сервер на prisma + faker; параллельная разработка |
| Нет доступа к тестовому ITSM-контуру | Высокая | Высокое | Запросить заранее; использовать статические тестовые данные |
| Изменение требований в процессе | Высокая | Среднее | Agile-процесс (спринты 2 недели); документирование изменений |
| Чат признан ОРИ (149-ФЗ) — доп. обязательства | Средняя | Высокое | Юридическая консультация; если ОРИ — регистрация, хранение 6 мес, СОРМ |
| Несоблюдение 152-ФЗ (штрафы, блокировка сайта) | Средняя | Критическое | Юрист с первой недели; compliance-гайд; локализация данных в РФ |
| Утечка ПДн (breach) | Низкая | Критическое | Шифрование, audit log, incident response plan, 24h notification РКН |
| Недоступность ITSM API (блокировка входа компаний) | Средняя | Высокое | Кэш данных с TTL + fallback stale cache |
| Сложность с биллингом для физлиц (post-MVP) | Низкая | Среднее | Вынести в отдельный эпик; ЮKassa / Tinkoff (они же решают 54-ФЗ) |
| Double-booking (два клиента на один слот) | Средняя | Высокое | Redis lock + optimistic locking в БД |
