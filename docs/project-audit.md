# Аудит проекта UCS Trainings

Дата аудита: 7 июля 2026 года.

## Краткий вывод

Сейчас репозиторий содержит рабочий по структуре frontend-прототип на Next.js 14 с App Router, TypeScript и Tailwind CSS. Пользовательские экраны, админские экраны, формы записи, форма отзыва и несколько API Route Handlers уже созданы, но почти вся бизнес-логика работает на статических JSON-файлах или только в памяти браузера. Полноценного backend, постоянной базы данных, Prisma/ORM и серверной авторизации нет.

Проект пока нельзя считать готовым MVP: данные не сохраняются, доступ к админке не защищён, формы не имеют полноценной серверной валидации, а конфигурация развёртывания зависит от ручной настройки корневой папки Vercel.

## 1. Что сейчас есть в проекте

### Структура папок

```text
ucs-trainins/
├── .github/
│   └── workflows/              # CI и два расписания вызова API
├── apps/
│   └── frontend/               # Next.js-приложение
│       ├── public/             # PWA manifest и иконки
│       ├── src/
│       │   ├── app/            # Страницы App Router и API Route Handlers
│       │   ├── components/     # UI-компоненты и layout
│       │   ├── data/           # Статические JSON-моки
│       │   ├── lib/            # Клиентский mock API и утилиты
│       │   ├── stores/         # Zustand-хранилища
│       │   └── types/          # TypeScript-типы
│       ├── .env.local.example
│       ├── package.json
│       ├── package-lock.json
│       ├── next.config.js

├── design-vars/                # Отдельные статические HTML-макеты
```

### Frontend

Frontend расположен в `apps/frontend`. Используются:

- Next.js 14.2 с App Router;
- React 18 и TypeScript;
- Tailwind CSS;
- Zustand для состояния в браузере;
- Zod присутствует в зависимостях, но в текущем коде не используется;
- `react-hot-toast` для уведомлений;
- `next-pwa` для manifest, service worker и PWA-сборки.

Реализованы страницы главной, услуг, специалистов, записи, списка записей, чата, уведомлений, профиля, входа, регистрации, отзыва, админского dashboard и админского расписания.

### Backend

Отдельного backend-приложения или сервиса нет. Частичный серверный слой находится внутри Next.js в `apps/frontend/src/app/api` и содержит Route Handlers:

- `POST /api/auth` — моковая проверка договора или безусловный вход физлица;
- `GET/POST /api/bookings` — чтение моковых записей и формирование ответа без сохранения;
- `GET /api/services`;
- `GET /api/specialists`;
- `GET /api/slots`;
- `GET/POST /api/reminders`;
- `POST /api/notify/email`;
- `POST /api/notify/telegram`.

Большинство страниц вообще не используют эти маршруты: `src/lib/api.ts` импортирует JSON напрямую, имитирует задержку и возвращает данные в браузере. Поэтому существующие Route Handlers не образуют полноценный backend.

Server Actions с директивой `use server` отсутствуют.

### База данных и ORM

Базы данных нет. Данные находятся в `apps/frontend/src/data/*.json`, а созданные записи и отзывы не записываются даже в эти файлы. Схемы данных, миграций и seed-механизма нет.

Prisma или другая ORM не установлены и не настроены. Папки `prisma`, файла `schema.prisma` и Prisma Client в проекте нет.

### Авторизация

Есть только интерфейс входа и регистрации и клиентское Zustand-состояние. Любые введённые значения фактически не проверяются страницами входа и регистрации. Состояние не сохраняется после обновления страницы, серверная сессия и cookie отсутствуют.

Нет NextAuth/Auth.js, middleware, хеширования паролей, восстановления пароля, ограничения попыток входа и серверной проверки ролей. Маршруты `/admin` и `/admin/schedule` доступны напрямую любому посетителю.

### Админка

Есть визуальный dashboard `/admin` и экран расписания `/admin/schedule`. Данные dashboard формируются в клиентском mock API, а кнопки расписания показывают уведомления, но не изменяют данные. Защиты доступа и полноценного CRUD нет.

### Формы заявок и записей

Есть многошаговая форма записи `/booking`, выбор услуги, специалиста, даты и слота. Подтверждение вызывает клиентский mock API и показывает успешное уведомление. Запись не сохраняется и после перехода отображается только заранее подготовленный JSON.

Формы входа и регистрации также являются демонстрационными: поля не связаны с серверной моделью, полноценной валидации и обработки ошибок нет.

### Отзывы

Есть статические отзывы в `src/data/reviews.json`, их отображение в админке и форма `/review`. Отправленный отзыв не сохраняется и не связан на сервере с пользователем, записью или специалистом.

### Деплой на Vercel

В `apps/frontend/vercel.json` заданы Next.js framework, `npm ci`, `npm run build` и каталог результата `.next`. В истории репозитория есть изменения, связанные с исправлением Vercel build, но в репозитории нет данных, подтверждающих текущий активный Vercel project или успешный production deployment.

Для развёртывания этого monorepo-подобного репозитория в настройках Vercel должен быть указан Root Directory `apps/frontend`. Если Root Directory оставить на корне репозитория, Vercel не найдёт корневой `package.json`, а вложенный `vercel.json` не будет применён как конфигурация проекта.

## 2. Что сейчас не так

### Чего не хватает для полноценного проекта

- постоянного хранения пользователей, услуг, специалистов, слотов, записей и отзывов;
- серверной авторизации и защищённых маршрутов;
- серверной бизнес-логики записи с проверкой доступности слота;
- административного CRUD;
- валидации входных данных и единообразной обработки ошибок;
- документации запуска, настройки окружения и деплоя;
- согласованного CI для lint, typecheck, Prisma и build;
- тестов критических пользовательских сценариев.

### Отсутствующие зависимости

Для выбранного в требованиях набора `Next.js + PostgreSQL + Prisma + Auth.js` отсутствуют Prisma Client, Prisma CLI, Auth.js/NextAuth и библиотека безопасного хеширования паролей. Для выполнения TypeScript seed-скрипта также потребуется runner TypeScript, если seed будет написан на TypeScript.

### Отсутствующие или несогласованные команды package.json

Сейчас есть `dev`, `build`, `start`, `lint` и `type-check`. Нет:

- команды с требуемым именем `typecheck` — есть только `type-check`;
- `prisma:generate`;
- `prisma:migrate`;
- `prisma:deploy`;
- `prisma:studio`;
- `prisma:seed`.

Prisma-команды нельзя добавлять до появления Prisma, схемы и seed-файла.

### Проблемы `apps/frontend`

- приложение смешивает два источника данных: клиентский `src/lib/api.ts` и Route Handlers в `src/app/api`;
- основной UI читает JSON напрямую через клиентский mock API, поэтому серверные маршруты почти не участвуют в работе продукта;
- Zustand auth хранится только в памяти и сбрасывается при обновлении страницы;
- mock-данные и тестовые учётные данные встроены в код;
- формы часто используют `defaultValue`, но не передают введённые значения в серверную логику;
- часть кнопок и экранов только имитирует действие;
- `allowJs` и `skipLibCheck` ослабляют проверку проекта, хотя сами по себе не блокируют MVP;
- PWA-плагин добавляет сложность сборке, но не решает offline-работу с реальными данными.

### Проблемы Vercel Root Directory

Корневого `package.json` нет. Следовательно, Vercel project должен быть явно привязан к `apps/frontend`. `apps/frontend/vercel.json` не исправляет неверно выбранный Root Directory, потому что при корне репозитория он находится не там, где Vercel ожидает конфигурацию проекта.

### Проблемы build

По структуре `package.json` команда build существует. Во время этого аудита build не запускался, потому что `node_modules` отсутствует, а задача прямо запрещает добавление зависимостей. Поэтому текущий build нельзя считать проверенным результатом аудита.

Дополнительные риски build и CI:

- CI вызывает `npm run type-check`, тогда как целевой публичный контракт из ТЗ требует `npm run typecheck`;
- `next lint` относится к текущей версии Next.js 14 и потребует пересмотра только при будущем обновлении Next.js;
- генерация Prisma Client должна выполняться до typecheck/build после добавления Prisma;
- корректная сборка на Vercel зависит от Root Directory `apps/frontend`.

### Проблемы env

Текущий `.env.local.example` содержит только `TELEGRAM_BOT_TOKEN`, `RESEND_API_KEY`, `APP_URL` и `API_SECRET`. Он не соответствует планируемому MVP и не содержит `DATABASE_URL`, `DIRECT_URL`, `AUTH_SECRET`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`, `NEXT_PUBLIC_APP_URL`.

Существующие GitHub Actions используют секреты `APP_URL` и `API_SECRET`, но API reminders не проверяет `Authorization`. Workflow cleanup вызывает отсутствующий `/api/cleanup`. Это создаёт ложное ощущение работающей автоматизации.

### Проблемы README

Корневой `README.md` пуст. Нет описания проекта, требований к Node.js, установки, запуска, команд, env, Prisma, структуры, Vercel Root Directory и процесса деплоя.

### Проблемы базы данных

- БД отсутствует;
- нет модели и ограничений целостности;
- нет уникальности email и защиты от двойного бронирования слота;
- нет миграций, seed и стратегии production migration;
- записи, отзывы, пользователи и расписание не сохраняются;
- нет разделения demo/test и production данных.

### Проблемы безопасности

- `/admin` и административные действия не защищены;
- endpoint `/api/auth` возвращает успешный вход физлица без проверки пароля;
- тестовый номер договора и данные пользователей находятся в исходном коде;
- endpoint создания записи принимает произвольный JSON без схемы валидации;
- endpoints email, Telegram и reminders не требуют авторизации и могут быть вызваны извне;
- нет rate limiting для входа и публичных форм;
- пароли не хешируются, потому что реального пользовательского хранилища нет;
- отсутствуют CSRF/session controls, аудит административных действий и контроль доступа к данным;
- CSP разрешает `'unsafe-inline'` и `'unsafe-eval'` для scripts; это снижает защиту от XSS;
- API внешних уведомлений не проверяет `res.ok`, поэтому ошибка провайдера может быть возвращена как обычный успешный HTTP-ответ;
- персональные данные демонстрируются в mock-файлах и логах; для production нужна политика минимизации и исключение чувствительных данных из логирования.

## 4. Какие зависимости нужны

Ниже перечислены зависимости, которые потребуется **добавить в следующем implementation PR**, а не в этом аудите.

### Production dependencies

| Зависимость | Назначение |
|---|---|
| `@prisma/client` | Типобезопасный доступ приложения к PostgreSQL по Prisma schema. |
| `next-auth` | Серверная авторизация Auth.js/NextAuth, сессии и Credentials provider для MVP. |
| `bcryptjs` | Хеширование и проверка паролей без хранения открытого пароля. |

`zod` уже установлен и должен использоваться для общей серверной и клиентской валидации. Отдельный validation package не нужен.

### Dev dependencies

| Зависимость | Назначение |
|---|---|
| `prisma` | Prisma CLI: генерация клиента, миграции, deploy миграций и Studio. |
| `tsx` | Запуск TypeScript seed-скрипта через `prisma db seed` без отдельной ручной компиляции. |

Новые библиотеки для UI, форм, запросов или state management на этапе MVP не обязательны: текущего стека достаточно.

## 5. Какие env-переменные нужны

| Переменная | Назначение |
|---|---|
| `DATABASE_URL` | Основная строка подключения Prisma к PostgreSQL. Для serverless/Vercel обычно используется pooled connection URL. |
| `DIRECT_URL` | Прямое подключение к PostgreSQL без pooler для Prisma migrations и административных операций. |
| `AUTH_SECRET` | Криптографический секрет Auth.js для подписи и защиты сессий и токенов. Должен быть длинным, случайным и разным по окружениям. |
| `ADMIN_EMAIL` | Email начального администратора, создаваемого seed-процессом. Не является заменой проверке роли в БД. |
| `ADMIN_PASSWORD` | Пароль начального администратора, который seed должен сразу хешировать. Нельзя коммитить или хранить в открытом виде. |
| `NEXT_PUBLIC_APP_URL` | Публичный базовый URL приложения для клиентских абсолютных ссылок и redirect URL. Значение доступно браузеру и не должно содержать секреты. |

Для локальной разработки значения должны находиться в `.env.local`, для Vercel — в Environment Variables, для GitHub Actions — в Secrets. Реальные значения не должны попадать в Git.

## 6. Какие команды должны быть в package.json

Целевой набор команд для `apps/frontend/package.json` после добавления Prisma:

| Команда | Ожидаемое действие |
|---|---|
| `npm run dev` | Запуск Next.js development server. Уже есть. |
| `npm run build` | Production build Next.js. Уже есть. |
| `npm run lint` | Запуск lint-проверки. Уже есть. |
| `npm run typecheck` | Запуск `tsc --noEmit`. Сейчас есть только команда `type-check`, имя нужно унифицировать. |
| `npm run prisma:generate` | Запуск `prisma generate`. |
| `npm run prisma:migrate` | Создание и применение локальной migration через `prisma migrate dev`. |
| `npm run prisma:deploy` | Применение уже созданных migration в Preview/Production через `prisma migrate deploy`. |
| `npm run prisma:studio` | Запуск Prisma Studio для локального просмотра данных. |
| `npm run prisma:seed` | Запуск настроенного seed через `prisma db seed`. |

Команды Prisma должны работать из `apps/frontend`, где будут находиться `package.json` и папка `prisma`.

## CI/CD правила

- Одна задача должна оформляться одним атомарным commit.
- Commit message должен иметь формат `type: short description`.
- Для этого аудита: `docs: add project audit`.
- Нельзя пушить код с неуспешными обязательными проверками.
- После появления Prisma CI должен выполнять install, Prisma generate, lint, typecheck и build.
- Код должен быть готов к deploy сразу после merge.
- Этот audit PR не должен мержиться автоматически и не содержит изменений приложения.
