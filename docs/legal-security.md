# Правовые и Security-требования

## 1. Законодательство РФ

### 1.1. 152-ФЗ "О персональных данных"

#### Основные принципы
- Обработка ПДн только на законной основе (ст.5)
- Минимизация собираемых данных
- Хранение не дольше, чем необходимо для целей обработки

#### Условия обработки (ст.6)
Основания для обработки ПДн в проекте:
- Согласие субъекта (ст.9) — для физлиц
- Исполнение договора (ст.6 ч.1 п.2) — для записи на услуги
- Законный интерес (ст.6 ч.1 п.5) — для аналитики (с ограничениями)

#### Согласие на обработку (ст.9)
Должно содержать:
- Наименование оператора
- Цель обработки
- Перечень обрабатываемых данных
- Перечень действий с данными
- Срок обработки
- Порядок отзыва согласия
- Подпись субъекта (accept checkbox)

**Реализация**: отдельный экран при регистрации с чекбоксом + ссылка на Политику.

#### Локализация данных (ст.18 ч.5, 242-ФЗ)
- Все ПДн граждан РФ должны обрабатываться в БД на территории РФ
- Запись, систематизация, накопление, хранение, уточнение, извлечение — только в РФ
- Нарушение: штраф до 6 млн руб. + блокировка сайта РКН

**Реализация**: PostgreSQL в Yandex Cloud (Москва/Владимир/Рязань).

#### Трансграничная передача (ст.12)
- Перечень стран с адекватной защитой: ЕАЭС, страны Совета Европы
- Для остальных: письменное согласие + контрактные условия
- При использовании SaaS (Sentry, GitHub) — оценить риски

**Реализация**: Использовать российские аналоги (Sentry → Yandex Appmetrica, GitHub → GitLab РФ).

#### Права субъекта (ст.14-17)
- Право на доступ (ст.14) → раздел "Мои данные"
- Право на исправление (ст.21) → редактирование профиля
- Право на удаление (ст.21) → "Удалить аккаунт"
- Право на отзыв согласия (ст.9) → toggle в настройках
- Срок ответа: 10 рабочих дней (ст.20)

#### Обязанности оператора (ст.18-22.1)
- Назначить ответственного за ПДн (ст.22.1)
- Утвердить Политику обработки ПДн (ст.18.1)
- Опубликовать Политику конфиденциальности на сайте (ст.18.1)
- Уведомить РКН до начала обработки (ст.22)
- Обеспечить безопасность ПДн (ст.19)

#### Уведомление об утечках (ст.21, 23.1)
- 24 часа — уведомление РКН
- 72 часа — полный отчёт об инциденте
- Незамедлительно — уведомление субъектов (если нарушены их права)
- Штрафы: до 6 млн руб. за повторное нарушение

#### Уровни защищенности (ПП РФ №1119)
**Для проекта: УЗ-3** (иные ПДн, < 100 000 субъектов, угрозы 1/2/3 типа)

Требования УЗ-3:
| Мера | Описание |
|------|----------|
| Идентификация и аутентификация | Пароль (min 8 символов), блокировка после N попыток |
| Управление доступом (RBAC) | Разграничение прав по ролям |
| Ограничение программной среды | Белый список разрешённых процессов |
| Антивирус | ClamAV на серверах |
| IDS/IPS | Система обнаружения вторжений |
| Регистрация событий | Аудит-лог всех действий с ПДн |
| Контроль целостности | Хэширование критичных файлов |
| Межсетевой экран | Микросегментация, запрет всего, кроме разрешённого |
| Криптография (СКЗИ) | По результатам модели угроз |
| Физическая защита | Дата-центр провайдера |

### 1.2. 149-ФЗ "Об информации"

#### Регистрация сайта
- ПП РФ №2467: владелец сайта обязан уведомить РКН
- Подаётся через портал РКН до начала обработки данных

#### Чат и ОРИ (ст.10.1)
**Риск**: Чат с отделом может быть признан ОРИ (Организатор Распространения Информации)

Обязанности ОРИ:
1. Регистрация в РКН
2. Хранение сообщений 6 месяцев (с 2024 г.)
3. Предоставление данных ФСБ по запросу
4. Установка ТС ОРМ/СОРМ (дорогостоящее оборудование)
5. Идентификация пользователей по номеру телефона

**Митигация**: 
- Консультация с юристом: является ли служебный чат в рамках оказания услуг ОРИ?
- Если да: регистрация ОРИ + хранение 6 мес + идентификация по телефону
- Если нет: документировать обоснование (чат в рамках услуги, а не публичная коммуникация)

#### Идентификация пользователей (ст.10.2-1)
- Для обмена сообщениями требуется идентификация
- Через номер телефона (SMS) или портал Госуслуг

Обязательные к реализации разделы:
- V2: Authentication (пароль, MFA, session management)
- V3: Session Management (httpOnly, Secure, SameSite cookies)
- V4: Access Control (RBAC, row-level)
- V5: Input Validation (whitelist, server-side)
- V6: Cryptography (TLS 1.3, secure key storage)
- V7: Error Handling (no stack traces in production)
- V8: Data Protection (encryption at rest, data classification)
- V9: Communications (TLS, HSTS, CSP)
- V11: Business Logic (double-booking prevention, rate limits)
- V12: File Uploads (type/size validation, malware scanning)

## 3. Технические требования безопасности

### 3.1. Transport Layer Security (TLS)

- TLS 1.3 (предпочтительно), TLS 1.2 (минимум)
- Отключить: TLS 1.0, 1.1, SSL 3.0
- Cipher suites:
  - TLS 1.3: `TLS_AES_256_GCM_SHA384`, `TLS_CHACHA20_POLY1305_SHA256`
  - TLS 1.2: `ECDHE-ECDSA-AES256-GCM-SHA384`, `ECDHE-RSA-AES256-GCM-SHA384`
- HSTS: `max-age=31536000; includeSubDomains; preload`
- OCSP Stapling: включено
- Perfect Forward Secrecy: обязательно (только ECDHE)

### 3.2. HTTP Security Headers

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' wss:; font-src 'self' data:; frame-ancestors 'none'; base-uri 'self'; form-action 'self'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 0
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

### 3.3. Authentication & Session Management

| Компонент | Требование |
|-----------|-----------|
| Пароль | min 8 символов (12+ рекомендуется), upper + lower + digit |
| Хэширование | argon2id (рекомендуется) или bcrypt (cost 12+) |
| MFA | TOTP (RFC 6238) для администраторов |
| Rate limiting | 5 попыток → 15 мин блокировка |
| Lockout | Экспоненциальная блокировка после N попыток |
| Session storage | httpOnly + Secure + SameSite=Strict cookies |
| Session TTL | 30 мин неактивности → logout |
| Refresh token | 7 дней, rotation на каждый refresh |
| Password reset | secure token (crypto.randomBytes), TTL 15 мин |
| Восстановление пароля | Email/SMS с одноразовой ссылкой |

### 3.4. Cookies

| Атрибут | Значение |
|---------|----------|
| HttpOnly | true (запрет доступа из JavaScript) |
| Secure | true (только по HTTPS) |
| SameSite | Strict (для API) / Lax (для страниц) |
| Path | / |
| Max-Age | Зависит от типа токена (15 мин access, 7 дней refresh) |

### 3.5. CSRF Protection

- SameSite=Strict cookies (основная защита)
- CSRF tokens для форм (double-submit cookie pattern)
- CORS: конкретный origin (не `*`), credentials: true при необходимости

### 3.6. File Upload Security

| Мера | Реализация |
|------|-----------|
| Whitelist типов | PDF, DOC, DOCX, XLS, XLSX, PNG, JPG, JPEG, GIF |
| MIME check | Проверка Content-Type + magic bytes |
| Размер | Макс 10 MB |
| Malware scan | ClamAV |
| Хранение | Вне webroot, UUID-имена, AES-256 шифрование |
| Purging | Очистка через 30 дней после последнего доступа |

### 3.7. API Security

| Мера | Реализация |
|------|-----------|
| Prefix | `/api/v1` |
| Auth | JWT в httpOnly cookie |
| Rate limit | 100 req/min/user, 30 req/min для auth |
| Input validation | class-validator DTO + whitelist: true |
| Body limit | 1 MB (10 MB для file upload) |
| Methods | Только GET, POST, PUT, PATCH, DELETE |
| Swagger | Только в dev/staging |

### 3.8. Audit Logging

**Что логировать:**
- Регистрация пользователя (email, phone, IP, user-agent)
- Вход/выход из системы (user_id, timestamp, IP)
- Изменение ролей (admin_action, before/after)
- Операции с ПДн (чтение, изменение, удаление)
- Неудачные попытки аутентификации
- Экспорт данных (кто, что, когда)
- Изменение настроек системы
- Операции с записями (создание, отмена, перенос)
- Вызовы API ITSM (request/response, latency)

**Хранение**: 
- Активные: 1 год (Loki / ELK)
- Архив: 3 года (сжатый, зашифрованный)
- Никогда не удалять — только архивировать

### 3.9. Secrets Management

| Secret | Хранение |
|--------|----------|
| DB password | Vault / Docker secrets / Yandex Lockbox |
| JWT secret | 256-bit random, rotation quarterly |
| ITSM API key | Encrypted in DB, HSM for production |
| SMTP credentials | Environment variables (encrypted) |
| Session secret | 256-bit random |
| Encryption keys | HSM / Vault |

### 3.10. Container Security

| Мера | Реализация |
|------|-----------|
| Base image | `node:20-alpine` (specific version, non-latest) |
| Image scanning | Trivy в CI/CD |
| User | Non-root (`node` user, UID 1000) |
| Filesystem | `readOnlyRootFilesystem: true` |
| Resource limits | CPU/memory limits |
| Secrets | Docker secrets, never in image |
| Capabilities | `--cap-drop=ALL --cap-add=NET_BIND_SERVICE` |
| Healthcheck | HEALTHCHECK instruction |
| Multi-stage build | Build → minimal runtime image |
| Network | Internal networks, no public DB/Redis |

## 4. Операционные требования

### 4.1. Резервное копирование

| Параметр | Значение |
|----------|----------|
| RPO (потеря данных) | 15 минут |
| RTO (восстановление) | 1 час (полное), 5 мин (read replica failover) |
| Full backup | Ежедневно (pgBackRest) |
| WAL | Каждые 5 минут |
| Retention | Daily: 30 дней, Weekly: 3 мес, Monthly: 1 год |
| Шифрование | GPG/AES-256 |
| Локация | Yandex Object Storage (РФ, другой регион) |
| Тест восстановления | Ежемесячно |

### 4.2. Мониторинг и алертинг

| Инструмент | Назначение |
|-----------|-----------|
| Sentry | Ошибки приложения (frontend + backend) |
| Prometheus | Метрики инфраструктуры (CPU, RAM, disk, network) |
| Grafana | Визуализация метрик, дашборды |
| Loki | Централизованный сбор логов |
| Alertmanager | Маршрутизация алертов (email, Telegram) |
| Uptime Kuma | Внешний мониторинг доступности |

**Ключевые алерты:**
- HTTP 5xx rate > 1% за 5 мин
- API response p95 > 500ms
- Disk usage > 80%
- Spike failed logins
- SSL certificate expiring (< 30 дней)
- Backup failure

### 4.3. Data Retention Policy

| Категория | Срок хранения | Действие после |
|-----------|---------------|----------------|
| Аккаунт (активный) | До отзыва согласия / удаления | Удалить в течение 30 дней |
| Аккаунт (неактивен 3 года) | auto-disable → уведомление → 30 дней | Soft delete → hard delete |
| Записи (bookings) | 3 года после услуги | Деперсонализация |
| Сообщения чата | 6 месяцев (ОРИ) | Удалить |
| Платёжные данные | 5 лет (налоговая) | Архив в отдельную БД |
| Audit logs | 1 год активный + 3 года архив | Архив → удаление |
| Session data | До expiry | Auto-expire в Redis |
| Email переписка | 1 год | Архив → удаление |
| Аналитика (деперсонализированная) | Бессрочно | — |

### 4.4. Incident Response Plan

1. **Detection** — Sentry alert / user report / monitoring
2. **Analysis** — determine scope, affected data, root cause
3. **Containment** — isolate affected systems, rotate secrets
4. **Notification**:
   - РКН: 24 часа (152-ФЗ)
   - Субъекты ПДн: незамедлительно (если их права нарушены)
   - Внутреннее: руководителю отдела
5. **Eradication** — remove threat, patch
6. **Recovery** — restore from backup
7. **Post-mortem** — RCA, update security measures, report

### 4.5. Business Continuity / Disaster Recovery

| Компонент | Стратегия |
|-----------|----------|
| Application | Min 2 инстанса в разных AZ |
| Database | PostgreSQL streaming replication (sync) |
| Redis | Sentinel / cluster |
| Failover | Auto (K8s) или manual (DNS) |
| DR site | Второй дата-центр РФ |
| DR test | Ежеквартально |
| DRT | 4 часа |
